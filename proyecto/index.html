1. Introducción

 

Un desarrollador de software, durante el proceso de desempeño de su trabajo (desarrollo de código fuente, desarrollo de aplicaciones, etc.) , por lo general suele encontrase con cierto tipo de problemas que lo dejan atascado o algunos errores que se niega a ser resuelto, no importa lo que se haga. En casos como este, los desarrolladores de software podrían necesitar respuestas a preguntas relacionadas con varios lenguajes de programación, plataformas de desarrollo, herramientas, APIs y servicios. La gran cuestión que se plantea es ¿Dónde se podría recurrir o consultar para dar solución a estas necesidades?.

Una de las posibles soluciones podría ser el de recurrir a sitios web o comunidades de desarrolladores informáticos, de tal modo que se puedan hacer preguntas o revisar las soluciones a todas sus dudas respecto al desarrollo de software. Una de las comunidades de desarrolladores informáticos en la web más popular se denomina “Stack Overflow”.

Stack Overflow. es la comunidad de desarrolladores informáticos en línea más grande y confiable de la web, en la que estos aprenden y comparten sus conocimientos. El sitio web sirve como una plataforma para que los usuarios puedan hacer y responder preguntas. Los usuarios registrados en la comunidad y que tienen participación activa en la comunidad, pueden votar preguntas y respuestas, editar preguntas y respuestas de manera similar a un Wiki.

Existen muchos sistemas exchange, entre ellos algunos bien conocidos Yahoo Answers y Quora, en el siguiente enlace se pueden ver más de estos sistemas. Stack Exchange.

Existen ya algunos estudios sobre estos sitios, y no solamente de temas de la industria de tecnología sino también de sistemas Q&A en general, por ejemplo sobre Yahoo Answers, el artículo denominado Measuring effectiveness and user satisfaction in Yahoo[1], que analiza 3 puntos: que tan rápido una pregunta es respondida, que tan rápido una pregunta es aceptada por el usuario por que satisface su interrogante y cuál es la relación entre una respuesta que satisface al usuario y el resto de respuestas, éste análisis tiene como objetivo determinar la eficiencia del sistema y la satisfacción del usuario.

Chirag et al [2] hace un estudio para predecir el tiempo de respuesta de una pregunta en StackOverflow, su análisis se basa en el análisis de características basadas en tags: como la popularidad del tag usado en una pregunta, tags populares, la probabilidad de que un conjunto de tags se muestren juntos en una pregunta, número de usuarios suscritos a los tags en base a las preguntas respondidas, y features correspondientes a la reputación de los usuarios y sus tiempos de respuesta, de estos features la reputación del usuario determinada por sus tiempos de respuestas era el feature que proporcionaba mayor fiabilidad al modelo, seguido de la popularidad del tag.

Mining Successful Answers et al [3], es otro de los artículos que estudia como predecir si una respuesta será marcada como aceptada, los factores estudiados son: Calidad de la presentación, afecto, tiempo y reputación o score de la respuesta, a diferencia de Chirag, en este estudio se evalua la calidad de la respuesta con análisis de sentimientos y con un modelo de regresión logística, obteniendo como resultado un mayor porcentaje de eficiencia en la feature de presencia de código en la respuesta, actitud positiva, reputación de la respuesta y del usuario que responde.

La base de datos de Stack OverFlow contiene gran cantidad de información sobre la cual crear distintos features para hacer un clasificador. En la propuesta del presente proyecto se analizará tanto las preguntas, respuestas y usuarios del sistema.

2. Descripción de los Datos

 

Los datos utilizados para el proyecto fueron extraídos de Stack Overflow Data , El servicio web de Google BigQuery permite realizar almacenamiento y consulta de conjuntos de datos masivos.

Para el proyecto se seleccionó la base de datos de Stack Overflow, está conformada de 16 tablas, de las cuales seleccionamos las más relevantes para el análisis

Tabla Registros
users 7'250.739
tags 49.306
post_questions 14'015.379
post_answers 22'046.899

Con la finalidad de crear un dataset para el proceso de análisis se extrajeron los datos de preguntas realizadas desde Enero de 2016 hasta Junio de 2017.

El data set resultante para el análisis contiene los siguientes atributos:

a_id, a_comment_count, a_community_owned_date, a_creation_date, a_last_activity_date, a_edit, a_last_editor_display_name, a_last_editor_user_id, a_owner_display_name, a_owner_user_id, a_parent_id, a_post_type_id, a_score, a_tags, q_id, q_accepted_answer_id, q_answer_count, q_comment_count, q_community_owned_date, q_creation_date, q_favorite_count, q_last_activity_date, q_last_edit_date, q_last_editor_display_name, q_last_editor_user_id, q_owner_display_name, q_owner_user_id, q_post_type_id, q_f0_, q_tags, q_view_count, c_accepted_answer_id, user_id, user_age, user_creation_date, user_reputation, user_up_votes, user_down_votes, user_views

Los datos mencionados anteriormente son el resultado de la consulta sobre las preguntas, respuestas y usuarios.

Para entender un poco más del significado de estos atributos, recurrimos a la Wiki de StackOverflow. Las preguntas, respuestas y usuarios tienen un puntaje o score que suma cuando un usuario da un voto y también puede disminuir, adicionalmente se puede observar el número de visitas, las fechas de creación y respuesta de la pregunta y reputación del usuario.

3. Procesamiento de Datos

 
  1. Se eliminaron los campos que no agregan valor a el análisis como id's y se reemplazaron los valores nulos, que eran datos vacíos por cero en los campos numéricos. 
     
select q_id, 
	case when a_comment_count is null then 0 else a_comment_count 
	end as a_comment_count,
	FORMAT_UTC_USEC(a_creation_date) as a_creation_date,
	a_edit, 
	case when a_score is null then 0 else a_score end as a_score, 
	case when  q_accepted_answer_id is null then 0 else 1 end as answer_acepted, 
	case when q_answer_count is null then 0 else q_answer_count end as q_answer_count, 
	case when q_comment_count is null then 0 else q_comment_count end as q_comment_count, 
	FORMAT_UTC_USEC(q_creation_date) as q_creation_date,
	case when q_favorite_count is null then 0 else q_favorite_count end as q_favorite_count, 
	q_tags,
	case when q_f0_ is null then 0 else q_f0_ end as q_score, 
	case when q_view_count is null then 0 else q_view_count end as q_view_count, 
	case when c_accepted_answer_id is null then 0 else 1 end as answer_selected, 
	case when user_reputation is null then 0 else user_reputation end as user_reputation, 
	case when user_up_votes is null then 0 else user_up_votes end as user_up_votes, 
	case when user_down_votes is null then 0 else user_down_votes end as user_down_votes, 
	case when user_views is null then 0 else user_views end as user_views, 
FROM 
	[fleet-automata-177802:filtros_stackoverflow.answer_question]

  2. Se agregaron features como la fecha de la primera respuesta a una pregunta.

     
select q_id, min(a_creation_date)from 
	[fleet-automata-177802:filtros_stackoverflow.answer_question] group by 1

  3. Se obtuvieron algunas estadísticas sobre el tiempo de respuesta de las preguntas aceptadas y y la primera respuesta de una pregunta.

    El comportamiento de la primera respuesta y la respuesta aceptada aparentemente es similar, esto indica que en la mayoría de los casos la primera respuesta tiene la solución acertada, sin embargo también se observa que hay preguntas que se responden después de días y en algunos casos meses. Este sería un factor determinante en determinar cuando una pregunta es más difícil que otra.

  4. Adicionalmente se agruparon los datos de preguntas y respuestas para crear una visualización multidimensional(Parallel Cordinates), luego de un proceso de normalización de datos, para descubrir patrones sobre la clasificación de las preguntas.

    Los datos están clasificados por las preguntas con respuestas aceptadas(1) y con respuestas no aceptadas(0). La visualización presenta mucho ruido debido a que las dos clases tienen un comportamiento similar, sin embargo si se pueden observar patrones como el score de una pregunta que para el caso de preguntas sin respuesta tienen un score menos de 0, esto puede deberse a la ambiguedad de la pregunta, que sería un factor a tener en cuenta en la clasificación. Además también se puede observar que para este set de datos que gran cantidad de respuestas aceptadas en este caso no corresponde a las primeras respuestas.

    5. Una vez limpiados los datos se realizaron algunos análisis para encontrar patrones y definir las features que se considerarán para el diseño del modelo.
